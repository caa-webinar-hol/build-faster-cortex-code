# Data Model: Financial Services CDC Source System

## Overview

Synthetic data generation app emulating a consumer banking / card processing source system running on SQL Server. Designed to produce realistic CDC patterns for the Openflow connector demo.

Inspired by real-world financial services CDC workloads observed in production Snowflake deployments.

---

## Source Database: `CardOpsDB`

A single SQL Server database with Change Tracking enabled, representing the operational backend of a consumer card issuer.

---

## Tables

### 1. `Transactions` — High-Volume INSERTs (the CDC firehose)

Card authorization events. This is the high-velocity table that demonstrates Openflow handling sustained insert volume.

| Column | Type | Description |
|--------|------|-------------|
| TransactionID | INT PK IDENTITY | Unique transaction identifier |
| AccountID | INT FK | Link to Accounts |
| MerchantID | INT FK | Link to Merchants |
| AuthDateTime | DATETIME2 | When the authorization occurred |
| Amount | DECIMAL(12,2) | Transaction amount |
| Currency | NVARCHAR(3) | Currency code (USD, EUR, etc.) |
| AuthDecision | NVARCHAR(20) | APPROVED, DECLINED, FLAGGED |
| DeclineReason | NVARCHAR(100) | Reason if declined (NULL if approved) |
| CardPresent | BIT | In-person vs. card-not-present |
| Channel | NVARCHAR(20) | POS, ONLINE, MOBILE, ATM |
| ResponseCode | NVARCHAR(10) | Network response code |
| CreatedAt | DATETIME2 | Record creation timestamp |

**CDC pattern:** ~70-80% of all generated events. Steady stream of INSERTs, no updates. High volume.

---

### 2. `Merchants` — Slow INSERTs + Occasional UPDATEs

Merchant reference data. New merchants appear occasionally; existing merchants get category or address corrections.

| Column | Type | Description |
|--------|------|-------------|
| MerchantID | INT PK IDENTITY | Unique merchant identifier |
| MerchantName | NVARCHAR(200) | Clean merchant name |
| CategoryCode | NVARCHAR(10) | MCC code |
| CategoryName | NVARCHAR(100) | Human-readable category (Groceries, Travel, Dining, etc.) |
| City | NVARCHAR(100) | Merchant city |
| State | NVARCHAR(50) | Merchant state |
| ZipCode | NVARCHAR(10) | Merchant zip |
| Country | NVARCHAR(3) | Country code |
| CreatedAt | DATETIME2 | Record creation timestamp |
| UpdatedAt | DATETIME2 | Last update timestamp |

**CDC pattern:** Slow. Occasional new merchant INSERTs when a transaction references a new merchant. Rare UPDATEs for category corrections or address changes.

---

### 3. `Accounts` — Slow INSERTs + UPDATEs (balance/status changes)

Customer card accounts. New accounts trickle in; balances and statuses update as transactions and payments process.

| Column | Type | Description |
|--------|------|-------------|
| AccountID | INT PK IDENTITY | Unique account identifier |
| AccountNumber | NVARCHAR(20) | Masked account number (last 4 visible) |
| AccountStatus | NVARCHAR(20) | ACTIVE, SUSPENDED, CLOSED |
| CreditLimit | DECIMAL(12,2) | Credit limit |
| CurrentBalance | DECIMAL(12,2) | Current balance |
| AvailableCredit | DECIMAL(12,2) | Available credit |
| OpenDate | DATE | Account open date |
| Region | NVARCHAR(50) | Customer region |
| RiskTier | NVARCHAR(20) | LOW, MEDIUM, HIGH, PREMIUM |
| CreatedAt | DATETIME2 | Record creation timestamp |
| UpdatedAt | DATETIME2 | Last update timestamp |

**CDC pattern:** Slow INSERTs for new accounts. Regular UPDATEs to CurrentBalance and AvailableCredit as transactions settle. Occasional status changes (ACTIVE → SUSPENDED).

---

### 4. `Disputes` — Medium INSERTs + Frequent UPDATEs (lifecycle progression)

Dispute cases that progress through a lifecycle. This is the table that best demonstrates CDC capturing mutations over time.

| Column | Type | Description |
|--------|------|-------------|
| DisputeID | INT PK IDENTITY | Unique dispute identifier |
| AccountID | INT FK | Link to Accounts |
| TransactionID | INT FK | Link to original transaction |
| DisputeAmount | DECIMAL(12,2) | Disputed amount |
| DisputeReason | NVARCHAR(100) | UNAUTHORIZED, DUPLICATE, NOT_RECEIVED, WRONG_AMOUNT |
| Status | NVARCHAR(30) | OPENED, INVESTIGATING, PROVISIONAL_CREDIT, RESOLVED, DENIED |
| FraudIndicator | BIT | Suspected fraud flag |
| MerchantName | NVARCHAR(200) | Merchant name (denormalized for operational use) |
| OpenedAt | DATETIME2 | When dispute was filed |
| LastUpdatedAt | DATETIME2 | Last status change |
| ResolvedAt | DATETIME2 | Resolution timestamp (NULL if open) |

**CDC pattern:** Medium INSERTs as disputes are filed. Frequent UPDATEs as cases progress through status lifecycle: OPENED → INVESTIGATING → PROVISIONAL_CREDIT → RESOLVED/DENIED.

---

### 5. `FraudAlerts` — Medium INSERTs (event stream)

Fraud detection events generated by the scoring engine. Insert-only event log, no updates.

| Column | Type | Description |
|--------|------|-------------|
| AlertID | INT PK IDENTITY | Unique alert identifier |
| TransactionID | INT FK | Link to flagged transaction |
| AccountID | INT FK | Link to account |
| AlertScore | DECIMAL(5,2) | Risk score (0-100) |
| AlertDecision | NVARCHAR(20) | PASS, FLAG, BLOCK |
| RuleTriggered | NVARCHAR(100) | Which rule fired (VELOCITY, GEO_ANOMALY, AMOUNT_SPIKE, etc.) |
| DeviceID | NVARCHAR(100) | Device fingerprint |
| IPAddress | NVARCHAR(50) | Source IP (masked for demo) |
| CreatedAt | DATETIME2 | When the alert was generated |

**CDC pattern:** Medium-volume INSERTs. No updates — these are immutable events. Volume correlates with Transactions table (a subset of transactions trigger alerts).

## Synthetic Data Notes

- **Transactions** reference existing Accounts and Merchants via FK
- **FraudAlerts** are generated for ~10-15% of transactions (those with FLAGGED or random sampling)
- **Disputes** reference existing Transactions — filed against recent transactions, progress through statuses over time
- **Accounts** balances should shift as transactions settle (CurrentBalance goes up, AvailableCredit goes down)
- **Merchants** use realistic MCC categories: Groceries (5411), Restaurants (5812), Gas Stations (5541), Online Retail (5999), Travel (4722), etc.
- All tables have Change Tracking enabled for Openflow CDC
